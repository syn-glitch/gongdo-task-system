## 이슈 목록

### #1 — 업무 등록 후 오류 피드백 부족

| 항목 | 내용 |
|------|------|
| **제보자** | 정혜림 |
| **심각도** | 🔴 HIGH |
| **피드백** | "등록했음에도 오류가 나서 등록되었는지 안되었는지 재차 확인 필요, 미등록이라면 다시 과정을 수행해야해 시간 낭비" |

**코드 분석:**

- `submitNewTask()` (`judy_workspace.html:3187~3229`)에서 성공 시 토스트 3초 표시 후 자동 사라짐
- 실패 시 `handleApiError()` 호출하지만, GAS `google.script.run`은 타임아웃 시 성공/실패 콜백 모두 안 올 수 있음
- 재시도 메커니즘(`retryWithExponentialBackoff`)이 등록에는 미적용 — 드래그&드롭 상태변경에만 적용 중
- 등록 중 로딩 인디케이터 없음 → 사용자가 "등록 중"인지 "멈춘 건지" 구분 불가

**판정:** 🐛 버그 + UX 결함

GAS 통신 특성상 타임아웃 발생 시 피드백이 아예 없어질 수 있으며, 재시도 로직도 없음.

**솔루션 제안:**

1. `submitNewTask()`에 로딩 스피너 + 버튼 비활성화 추가
2. 등록 API에도 재시도 패턴 적용 (`google.script.run` 직접 체이닝 방식으로)
3. 타임아웃 감지 시 "등록이 불확실합니다. 확인 후 재시도하세요" 안내 메시지 제공
4. 등록 성공 후 토스트 대신 등록된 항목 하이라이트 등 시각적 확인 강화

### #2 — 카테고리 확장 및 직접 입력란 부재

| 항목 | 내용 |
|------|------|
| **제보자** | 정혜림 |
| **심각도** | 🟡 MEDIUM |
| **피드백** | "새업무 등록 시, 카테고리 확장 및 직접 입력란이 필요함" |

**코드 분석:**

- 현재 "카테고리" 개념 자체가 없음 — 프로젝트(Project) select만 존재 (`html:3169~3185`)
- 프로젝트 목록은 `getProjectOptionsForWeb()`으로 "Projects" 시트에서 동적 로드 (`web_app.gs:599~615`)
- 드롭다운만 제공, 직접 입력(텍스트) 불가
- 기본값: `"기본 프로젝트"` 1개

**판정:** 📐 기능 부재

사용자가 원하는 "카테고리"는 프로젝트와 별개 개념일 수 있음 (예: 업무 유형, 긴급도 등). 프로젝트 외 분류 체계가 없고, 신규 항목 추가도 불가.

**솔루션 제안:**

1. 프로젝트 드롭다운에 "직접 입력" 옵션 추가 → 텍스트 입력 필드 토글
2. 별도 카테고리/태그 필드 신설 검토 (업무 유형: 기획, 디자인, 개발, 운영 등)
3. 백엔드 Projects 시트에 사용자가 직접 추가할 수 있는 경로 마련

### #4 — 내 업무/칸반 데이터 중복 문제

| 항목 | 내용 |
|------|------|
| **제보자** | 이지은 |
| **심각도** | 🟡 MEDIUM |
| **피드백** | "연동되긴 해도, 내용이 중복되어 사용하면서 무엇을 중점으로 해서 활성화할지 살펴볼 필요. 시각화는 칸반보드가 더 나은 듯함" |

**코드 분석:**

- 내 업무 테이블 (`html:2991~3051`): `getMyTasksForWeb()` API → `currentTasks` 변수
- 칸반 보드 (`html:3339~3351`): `getAllTasksForWeb()` API → `kanbanTasks` 변수
- 서로 다른 API 호출, 같은 데이터를 중복 로드
- 두 탭이 동일한 업무를 다른 형태로 보여주되, 명확한 차별화가 부족

**판정:** 📐 정보 설계(IA) 개선 필요

같은 데이터가 테이블 뷰와 칸반 뷰로 중복 표현되어 사용자 혼란 발생. 각 뷰의 목적이 명확하지 않음.

**솔루션 제안:**

1. **뷰 전환 방식** 채택: "내 업무" 탭 안에서 테이블↔칸반 토글 (중복 탭 제거)
2. 또는 각 뷰의 목적 차별화: 테이블 = 상세 필터/정렬, 칸반 = 상태 기반 드래그 관리
3. 데이터 소스 통합: 하나의 API로 로드 후 뷰만 분리 (일관성 확보)

### TD-001 — 프론트엔드 업무 API 중복 호출 통합

| 항목 | 내용 |
|------|------|
| **TD ID** | TD-001 |
| **관련 이슈** | 김감사 #4 (내 업무/칸반 데이터 중복) |
| **우선순위** | 🟡 Medium (2주 이내) |
| **유형** | refactoring |
| **담당** | 🏗️ 리팩터 |
| **상태** | Backlog (분석 완료, 구현 대기) |

**현황 분석 (Before)**:

프론트엔드에서 동일 데이터를 3회 별도 API 호출:

| 호출 위치 | API 함수 | 용도 | 코드 위치 |
|-----------|---------|------|----------|
| 내 업무 탭 | `getMyTasksForWeb()` | 내 업무 필터링 | `html:2934` |
| 칸반 탭 | `getAllTasksForWeb()` | 전체 업무 칸반 | `html:3351` |
| 캘린더 탭 | `getAllTasksForWeb()` | 전체 업무 캘린더 | `html:3594` |

백엔드 구조:
- `getAllTasksForWeb(userId)` → CacheService 5분 TTL 적용 (`web_app.gs:357`)
- `getMyTasksForWeb(userId)` → 내부에서 `getAllTasksForWeb()` 호출 후 필터링 (`web_app.gs:416`)
- 캐시 덕분에 시트 중복 읽기는 없으나, `google.script.run` GAS 통신 오버헤드(~500ms/회)가 3회 발생

**리팩토링 설계 (After)**:

```
[현재] 탭 전환 시마다 별도 API 호출
  내 업무 → google.script.run.getMyTasksForWeb() → currentTasks
  칸반   → google.script.run.getAllTasksForWeb() → kanbanTasks
  캘린더 → google.script.run.getAllTasksForWeb() → calendarEvents

[개선] 앱 초기화 시 1회 호출 → 공유 데이터
  초기화 → google.script.run.getAllTasksForWeb() → window.sharedTaskData
  내 업무 → sharedTaskData.filter(t => t.assignee === userName)
  칸반   → sharedTaskData (직접 사용)
  캘린더 → sharedTaskData (직접 사용)
  갱신   → 데이터 변경 이벤트 시 1회 재호출 → 모든 뷰 동기 갱신
```

**기대 효과**:

| 항목 | Before | After | 개선 |
|------|--------|-------|------|
| API 호출 횟수 (초기 로드) | 3회 | 1회 | -67% |
| GAS 통신 오버헤드 | ~1.5초 | ~500ms | -67% |
| 데이터 일관성 | 탭별 독립 (타이밍 차이 가능) | 단일 소스 (항상 동기) | 일관성 확보 |

**부작용 분석**:
- 내 업무 필터링이 클라이언트로 이동 → 필터 로직 정확성 검증 필요
- `sharedTaskData` 갱신 시점 설계 필요 (업무 등록/상태 변경/드래그 후)
- 백엔드 `getMyTasksForWeb()` 함수는 유지 가능 (다른 곳에서 호출 시 대비)

**구현 위임**: 🤵 자비스 개발팀 (프론트엔드 데이터 관리 구조 변경)
