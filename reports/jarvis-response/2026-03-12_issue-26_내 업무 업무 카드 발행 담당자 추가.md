## QA 보고서 핵심 요약
김감사팀의 분석 결과, 다음 3가지 핵심 이슈가 확인되었습니다:

1. **복수 담당자 지정 불가**: 현재 업무 카드는 1명의 담당자만 지정 가능하여 공동 작업이 필요한 업무 처리가 비효율적
2. **참조자 기능 부재**: 업무 내용을 함께 알아야 하는 팀원을 참조로 추가할 수 없어 정보 공유 단절
3. **협업 워크플로우 제약**: 팀 단위 업무 관리에 필수적인 기능 누락으로 협업 도구로서의 경쟁력 저하

## 원인 추정

### 데이터 모델 레벨
- **파일**: `backend/models/Task.gs` (추정)
- **원인**: 업무 카드 스키마가 `assignee` (단일 문자열) 구조로 설계
- **구체적 설명**: 
  - 현재 구조: `{ assignee: "user_id" }`
  - 배열 구조 미지원으로 복수 담당자 저장 불가
  - 참조자(`watchers`) 필드 자체가 존재하지 않음

### API 레벨
- **파일**: `backend/api/TaskAPI.gs` (추정)
- **함수**: `createTask()`, `updateTask()` (추정)
- **원인**: 
  - POST/PATCH 엔드포인트가 단일 담당자 ID만 처리
  - 배열 형태의 담당자 목록 파싱 로직 부재

### UI 컴포넌트 레벨
- **파일**: `frontend/components/TaskModal.html` (추정)
- **함수**: 담당자 선택 드롭다운 초기화 함수 (추정)
- **원인**: 
  - Single-select 드롭다운으로 구현
  - "담당자 추가" 버튼 및 태그 표시 UI 컴포넌트 부재

## 영향 범위

### 직접 영향
- **업무 생성/수정 플로우**: 담당자 지정 방식 전면 개편 필요
- **데이터베이스 스키마**: 기존 단일 담당자 데이터 마이그레이션 필요

### 연쇄 영향
- **알림 시스템** (`NotificationService.gs`): 복수 담당자에게 알림 발송 로직 추가
- **대시보드** (`Dashboard.html`): 담당자별 업무 집계 쿼리 수정 (`assignee` → `assignees.includes()`)
- **검색/필터** (`SearchFilter.gs`): 담당자 필터링 로직 배열 처리 방식으로 변경
- **권한 관리** (`PermissionService.gs`): 복수 담당자의 편집 권한 검증 로직 추가
- **모바일 뷰**: 복수 담당자 표시 레이아웃 재설계

## 수정 계획

### 1. 데이터 모델 변경 (Phase 1)
- `Task` 스키마 수정
  - `assignee` → `assignees: []` (배열)
  - `watchers: []` 필드 추가 (참조자)
- 마이그레이션 스크립트 작성
  - 기존 `assignee` 값을 `assignees[0]`으로 변환
  - 빈 `watchers` 배열 초기화

### 2. API 수정 (Phase 1)
- `TaskAPI.gs` 엔드포인트 수정
  - `createTask()`: `assignees` 배열 파싱 및 유효성 검사
  - `updateTask()`: 담당자 추가/제거 로직 구현
  - `addWatcher()`, `removeWatcher()` 신규 메서드 추가

### 3. UI 컴포넌트 개발 (Phase 2)
- Multi-select 드롭다운 구현
  - 담당자 검색 자동완성
  - 선택된 담당자 태그 형태 표시
  - 담당자 제거 (X 버튼)
- "참조자 추가" 섹션 신규 개발
  - 담당자와 구분되는 UI (회색 배지)
  - 읽기 전용 권한 표시

### 4. 연관 기능 수정 (Phase 3)
- 알림 시스템: 담당자 배열 순회 알림 발송
- 대시보드: `assignees` 배열 기반 집계 쿼리
- 검색/필터: `assignees.includes(userId)` 조건 처리
