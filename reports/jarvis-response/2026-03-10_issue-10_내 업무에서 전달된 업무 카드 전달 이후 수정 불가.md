## QA 보고서 핵심 요약
1. **권한 로직 오류**: 업무 전달 후 원작성자(전달자)의 수정 권한이 자동으로 제거되어 읽기 전용 모드로 전환됨
2. **업무 프로세스 차단**: 잘못된 업무 지시 후 수정/삭제가 불가능하여 실무 협업에 치명적 영향
3. **소유권 이전 구조 문제**: 전달 시 소유권이 수신자에게 완전히 이전되면서 전달자의 권한이 박탈되는 것으로 추정

## 원인 추정
**추정 파일 및 함수:**
- **프론트엔드**: `Code.gs` 내 `showTaskModal()` 함수의 권한 체크 로직
- **데이터 레이어**: `TaskService.gs` 또는 `DataService.gs`의 `canEditTask()` 권한 검증 함수
- **스프레드시트**: `업무 목록` 시트의 `상태(status)` 컬럼 또는 `소유자(owner)` 컬럼 참조 로직

**구체적 원인:**
1. 업무 전달 시 `전달자(creator)` 필드는 유지되지만, 편집 권한 체크 시 `담당자(assignedTo)` 필드만 확인하는 로직
2. GAS HTML 템플릿에서 `task.status === 'delivered'` 상태일 때 무조건 읽기 모드로 렌더링
3. `canEdit` 권한 체크에서 `Session.getActiveUser().getEmail() !== task.assignedTo` 조건만 사용하여 원작성자 예외 처리 누락

```javascript
// 추정 코드 (현재 버그 코드)
function canEditTask(taskId) {
  var task = getTaskById(taskId);
  var currentUser = Session.getActiveUser().getEmail();
  
  // 문제: 담당자만 확인하고 전달자(creator) 체크 누락
  return task.assignedTo === currentUser;
}
```

## 영향 범위
1. **직접 영향**: 업무 전달 기능을 사용하는 모든 사용자 (특히 관리자/팀장)
2. **간접 영향**: 
   - 업무 댓글/첨부파일 수정 기능도 동일 권한 로직을 사용할 경우 같은 문제 발생 가능성
   - 팀 업무 전달 시에도 동일 이슈 존재 가능
3. **비즈니스 영향**: 
   - 잘못된 업무 지시 수정을 위해 슬랙/이메일 등 외부 채널 사용 강제
   - 시스템 신뢰도 저하 및 사용자 불만 증가

## 수정 계획
1. **권한 로직 수정** (`TaskService.gs` 또는 `DataService.gs`)
   ```javascript
   function canEditTask(taskId) {
     var task = getTaskById(taskId);
     var currentUser = Session.getActiveUser().getEmail();
     
     // 수정: 담당자 OR 전달자(creator) 체크 추가
     return task.assignedTo === currentUser || 
            task.creator === currentUser;
   }
   ```

2. **UI 조건부 렌더링 수정** (`TaskModal.html`)
   - 읽기 전용 모드 조건에서 `task.creator === currentUser` 예외 추가
   - 편집 버튼 표시 로직에 전달자 권한 반영

3. **전달 취소 기능 추가** (중기 개선)
   - 전달된 업무를 다시 자신에게 되돌리는 "전달 취소" 버튼 구현
   - 전달 후 24시간 이내로 제한 조건 추가

4. **업무 상태 구조 개선**
   - `status` 필드에 `delivered` 상태 추가 시 권한 매트릭스 문서화
   - 스프레드시트에 `전달_이력` 시트 추가 (전달자, 수신자, 전달일시 기록)

5. **UX 개선**
   - 전달 확인 다이얼로그에 "전달 후에도 수정 가능합니다" 안내 문구 추가
   - 읽기 모드 진입 시 "전달자는 수정 가능" 툴팁 표시
