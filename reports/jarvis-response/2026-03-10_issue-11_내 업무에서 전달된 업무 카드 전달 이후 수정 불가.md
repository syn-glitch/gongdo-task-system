## QA 보고서 핵심 요약
1. **권한 로직 오류**: 업무 전달 시 송신자의 편집 권한이 완전히 박탈되어 전달 후 수정/삭제 불가
2. **워크플로우 중단**: 잘못된 업무 지시 시 정정 수단이 없어 업무 효율성 심각한 저하
3. **협업 시나리오 미고려**: 공동 편집, 회수, 재전달 등의 실무 상황을 고려하지 않은 설계

## 원인 추정
- **파일**: `Code.gs` (업무 전달 처리 로직) 및 `TaskCard.html` (UI 렌더링)
- **함수**: 
  - `delegateTask()` 또는 `transferTask()` — 업무 전달 처리 함수
  - `updateTaskOwnership()` — 소유권 변경 함수
  - `checkEditPermission()` — 권한 체크 함수
- **원인**: 
  1. 업무 전달 시 `task.owner` 필드를 수신자로 완전 변경하면서 송신자 권한을 제거하는 로직
  2. 권한 체크 로직이 `task.owner === currentUser`로만 구현되어, 송신자(delegator) 정보가 별도 관리되지 않음
  3. UI 렌더링 시 `isEditable` 플래그가 소유자(owner) 여부만으로 판단되어 전달자(delegator)에게도 읽기 전용으로 표시됨

## 영향 범위
- **직접 영향**: 모든 업무 전달 기능 사용자 (특히 협업이 잦은 팀)
- **간접 영향**: 
  - 업무 전달 기능의 신뢰도 하락으로 사용률 감소 가능
  - 잘못된 전달 시 Slack/이메일 등 외부 커뮤니케이션 비용 증가
  - 긴급 상황(오타, 잘못된 담당자 지정)에서 업무 지연 발생
- **데이터 정합성**: 현재는 영향 없으나, 향후 협업 기능 확장 시 병목 가능성

## 수정 계획
1. **데이터 모델 개선**
   - `Task` 객체에 `delegator` 필드 추가 (전달자 정보 보존)
   - `status` 필드에 `DELEGATED` 상태 추가 (기존 `ASSIGNED`와 구분)
   - 전달 이력 추적을 위한 `delegationHistory` 배열 추가 (선택)

2. **권한 로직 수정**
   - `checkEditPermission()` 함수 수정: `task.owner === user || task.delegator === user` 조건으로 변경
   - 전달 취소(Recall) 함수 신규 구현: `recallDelegation()` — 수신자가 업무를 시작하지 않았을 때만 가능하도록 조건 추가

3. **UI 개선**
   - `TaskCard.html`에 전달 취소 버튼 추가 (조건부 렌더링)
   - 전달 확인 다이얼로그에 경고 메시지 추가: "전달 후에도 수정 가능하나, 수신자가 확인 후에는 취소할 수 없습니다."
   - 읽기 전용 상태 시 안내 문구 개선

4. **테스트 케이스 추가**
   - 전달 후 송신자 수정 가능 여부
   - 전달 취소 후 소유권 복원 확인
   - 수신자 확인 후 취소 불가 확인
