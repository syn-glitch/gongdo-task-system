## 1. 역할 분리 원칙

### ✅ 필수 규칙: "Stay in Your Lane"

각 QA 전문가는 **자신의 전문 영역에만 집중**하고, 다른 영역은 코멘트하지 않습니다.

| 역할 | 검증 대상 | ❌ 금지 사항 |
|------|----------|-----------|
| **🔍 테스터** | 기능 로직, 에러 핸들링, API 응답 | 보안 취약점, UI 디자인 |
| **🛡️ 보안 감사관** | 인증, 권한, API 키, 동시성 | 기능 로직, UI/UX |
| **🎨 UX 검증관** | UI 일관성, 반응형, 접근성 | 백엔드 로직, 보안 |
| **🕵️‍♂️ 김감사 (QA Lead)** | 전체 통합, 최종 승인/반려 | - |

### 예시

#### ✅ 좋은 예 (테스터)
```markdown
## 발견된 문제
- ❌ API 응답 검증이 누락됨 (Line 451)
- ❌ null 체크가 없어 크래시 가능성 있음 (Line 62)
```

#### ❌ 나쁜 예 (테스터가 역할 벗어남)
```markdown
## 발견된 문제
- ❌ API 응답 검증이 누락됨 (Line 451)
- ❌ API 키가 하드코딩되어 보안 위험 있음 (Line 447) ← 보안 감사관 영역
- ❌ 버튼 색상이 다크모드에서 가독성 낮음 (Line 85) ← UX 검증관 영역
```

## 2. 리뷰 시간 제한

**철학**: "완벽한 QA보다 빠른 피드백"

### ⏱️ 시간 제한 (엄격)

| 역할 | 제한 시간 | 초과 시 |
|------|----------|--------|
| **🔍 테스터** | 15분 | 10분 내 주요 이슈만 리포트 제출 |
| **🛡️ 보안 감사관** | 10분 | 7분 내 주요 이슈만 리포트 제출 |
| **🎨 UX 검증관** | 10분 | 7분 내 주요 이슈만 리포트 제출 |
| **🕵️‍♂️ 김감사 (통합)** | 10분 | 8분 내 통합 리포트 제출 |

### 시간 단축 팁

1. **체크리스트 활용**: 템플릿의 체크리스트를 먼저 확인
2. **Critical/High 우선**: 낮은 우선순위는 시간 남으면 추가
3. **파일 범위 제한**: 변경된 파일만 집중 검토
4. **자동화 도구**: Grep, Glob으로 패턴 검색

## 3. 리포트 작성 규칙

### 3-1. 파일명 규칙 ✅ 필수

```
YYYY-MM-DD_feature_name_{functional|security|ux}_qa.md
```

**예시**:
- `2026-02-26_slack_message_action_functional_qa.md`
- `2026-02-26_judy_drag_drop_security_audit.md`
- `2026-02-26_kanban_board_ux_validation.md`

---

### 3-2. 리포트 구조 ✅ 필수

**모든 QA 리포트는 다음 섹션을 포함해야 합니다:**

```markdown
# [역할] 기능명 QA 리포트

**QA 담당**: [이름]
**검수일**: YYYY-MM-DD
**대상 파일**: [파일 경로]
**우선순위**: 🔴/🟠/🟡/🟢

---

## 📋 요약 (Executive Summary)
- 전체 평가 1-2문장

## ✅ 통과 항목 (Passed)
- 정상 동작하는 항목 나열

## ❌ 발견된 문제 (Issues Found)
| 우선순위 | 위치 | 문제 | 수정안 |
|---------|------|------|--------|
| 🔴 Critical | Line 447 | ... | ... |

## ⚠️ 개선 제안 (Recommendations)
- 선택 사항이지만 개선하면 좋을 항목

## 🎯 우선순위 분류
- 🔴 Critical: X개
- 🟠 High: X개
- 🟡 Medium: X개
- 🟢 Low: X개

## 📝 최종 판정
- ✅ 승인 / ❌ 반려 / ⚠️ 조건부 승인
```

---

### 3-3. 우선순위 정의 ✅ 필수

| 우선순위 | 정의 | 예시 |
|---------|------|------|
| 🔴 **Critical** | 기능이 전혀 작동하지 않음, 데이터 손실, 보안 취약점 | API 키 노출, 크래시, 인증 우회 |
| 🟠 **High** | 주요 기능에 심각한 영향, 사용자 경험 매우 나쁨 | 에러 핸들링 누락, 모바일 미지원 |
| 🟡 **Medium** | 기능은 작동하나 불편함, 일부 케이스 오작동 | Edge Case 미처리, 느린 응답 |
| 🟢 **Low** | 매우 작은 문제, 코드 품질 개선 | 변수명 개선, 주석 추가, 로그 개선 |

## 4. 승인 기준

### 4-1. 승인/반려 기준 ✅ 필수

| 상태 | 조건 | 조치 |
|------|------|------|
| **✅ 승인** | Critical/High 이슈 **0개** | 즉시 배포 가능 |
| **⚠️ 조건부 승인** | Medium 이슈만 있음 | 배포 후 추적 관리 |
| **❌ 반려** | Critical/High 이슈 **1개 이상** | 수정 후 재검토 필요 |

### 4-2. 김감사 (QA Lead) 최종 판정

**김감사는 3개 리포트를 통합하여 최종 결정합니다:**

```
┌─────────────────────────────────────────────┐
│  입력: 3개 리포트                            │
│  • functional_qa.md                         │
│  • security_audit.md                        │
│  • ux_validation.md                         │
└─────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────┐
│  통합 검토                                   │
│  • 중복 이슈 제거                            │
│  • 우선순위 재조정                           │
│  • Critical/High 개수 확인                  │
└─────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────┐
│  최종 판정                                   │
│  • ✅ 승인                                  │
│  • ⚠️ 조건부 승인 (Medium만 있음)           │
│  • ❌ 반려 (Critical/High 있음)             │
└─────────────────────────────────────────────┘
```

## 5. 커뮤니케이션 규칙

### 5-1. QA 팀 내부 소통 ❌ 금지

**QA 전문가들은 서로 직접 소통하지 않습니다.**

- ❌ 테스터가 보안 감사관에게 "이거 보안 문제 아닌가요?" 질문
- ❌ UX 검증관이 테스터에게 "이 기능 작동 안 돼요" 보고

**이유**: 각자 독립적으로 리포트를 작성해야 편향 없는 다각도 분석 가능

---

### 5-2. 김감사 (QA Lead)와의 소통 ✅ 허용

**모든 질문/보고는 김감사에게만 합니다.**

- ✅ 테스터 → 김감사: "이 파일이 변경 범위에 포함되나요?"
- ✅ 보안 감사관 → 김감사: "API 키 관리 정책이 뭔가요?"
- ✅ UX 검증관 → 김감사: "디자인 가이드라인 문서 어디 있나요?"

---

### 5-3. 파일 시스템을 통한 비동기 소통 ✅ 권장

**문서를 읽고 이해하세요:**

QA 리뷰 시작 전 다음 문서를 읽습니다:
1. `planning/implementation_plans/` - 개발 계획서
2. `docs/architecture/` - 아키텍처 문서
3. `docs/guides/AI_AGENT_TEAM_RULES.md` - 팀 규칙

## 6. 에스컬레이션 프로세스

### 6-1. 심각한 문제 발견 시 🚨

**Critical 이슈 발견 즉시 김감사에게 보고:**

```markdown
🚨 **Critical 이슈 발견**

**파일**: src/gas/slack_command.gs:447
**문제**: API 키가 하드코딩되어 GitHub에 노출됨
**영향**: 모든 사용자 데이터 접근 가능
**조치**: 즉시 배포 중단 및 키 재발급 필요
```

김감사는 즉시:
1. 팀장(USER)에게 보고
2. 배포 중단 요청
3. 긴급 수정 지시

---

### 6-2. 의견 충돌 시 🤝

**예시**: 테스터는 "승인", 보안 감사관은 "반려"

**해결 프로세스**:
1. 김감사가 두 리포트 읽고 판단
2. 우선순위 재평가 (보안 > 기능)
3. 최종 결정 및 팀장 보고

---

## 7. 특수 상황 규칙

### 7-1. Fast-Track 프로세스 (Micro 작업)

**예상 소요 시간 < 30분인 작업**:

- **QA 팀 축소**: 테스터 1명만 리뷰
- **시간 제한**: 5분
- **리포트 간소화**: 체크리스트만 확인

**예시**: 오타 수정, CSS 미세 조정

---

### 7-2. 긴급 핫픽스

**운영 서버 긴급 수정 (Production Hotfix)**:

- **QA 팀**: 전체 (3명)
- **시간 제한**: 각 5분 (병렬)
- **승인 기준**: Critical 이슈 0개만 확인

## 8. QA 도구 활용

### 8-1. 코드 검색 도구

```bash
# API 키 검색
grep -r "xoxb-" src/

# 에러 핸들링 검색
grep -r "try {" src/ | wc -l
grep -r "catch" src/ | wc -l

# TODO 주석 검색
grep -r "TODO" src/
```

### 8-2. 템플릿 활용

**QA 시작 전 템플릿 복사:**

```bash
cp qa/templates/functional_qa_template.md qa/qa_reviews/functional/2026-02-26_feature_name_functional_qa.md
```

## 9. 회고 참여 규칙

### 9-1. 정기 회고 (김감사 주도)

**QA 팀 전체 참여 (선택적)**:

- **일간 회고 (5분)**: QA 팀 리더만 (김감사)
- **주간 회고 (30분)**: QA 팀 전체 참여 권장
- **월간 회고 (1시간)**: QA 팀 전체 필수 참여

### 9-2. 회고 주제

- 발견한 버그 패턴 분석
- QA 프로세스 개선 제안
- 팀 협업 문제점
- 새로운 QA 기법 학습
